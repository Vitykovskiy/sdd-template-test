# select_task.py

## Назначение

Скрипт выбирает ровно одну исполнимую задачу из GitHub Project. Это единственный допустимый механизм выбора активной задачи в режиме `issue_driven`.

## Контекст применения

Используется routing-инструкцией стадии `issue_driven`. Ручной выбор задачи в обход этого скрипта запрещён.

## Входные параметры

- `--state-file`
  - путь к `.codex/state.json` относительно корня репозитория;
  - по умолчанию: `.codex/state.json`.

## Внешние зависимости

- `gh`
- `.codex/state.json`

## Какие задачи считаются валидными

Задача должна одновременно удовлетворять условиям:

1. Контент project item должен быть типа `Issue`.
2. Должен быть ровно один role label из поддерживаемого множества.
3. Должен быть ровно один priority label.
4. В body должны присутствовать обязательные секции:
   - `Id`
   - `Role`
   - `Title`
   - `Purpose`
   - `Artifacts`
   - `Priority`
5. Значения в body должны совпадать с labels там, где это применимо.
6. `project Status` должен совпадать с workflow status задачи.
   Канонический набор workflow status: `ready`, `in_progress`, `review`, `in_testing`, `done`.

## Алгоритм работы

1. Находит корень репозитория по наличию `.codex/state.json`.
2. Загружает состояние и читает `project.gh_project_url`.
3. Парсит URL GitHub Project.
4. Через `gh project item-list` получает все items проекта.
5. Для каждого item:
   - валидирует метки и структуру body;
   - формирует список нарушений для невалидных задач;
   - добавляет валидные задачи со статусом `ready` в пул кандидатов.
6. Если валидных `ready`-задач нет, возвращает `no_ready_task`.
7. Если валидные задачи есть, выбирает одну по deterministic policy:
   - сначала по приоритету `critical -> high -> medium -> low`;
   - затем по `task_id` в лексикографическом порядке.
8. Возвращает JSON с выбранной задачей и списком невалидных задач.

## Формат результата

Основные статусы:

- `selected`
- `no_ready_task`
- `configuration_blocker`
- `github_cli_error`

Основные поля:

- `project_url`
- `selected_task`
- `invalid_tasks`
- `reason`

## Коды завершения

- `0` — задача выбрана.
- `2` — валидной `ready`-задачи нет.
- `3` — конфигурационный блокер.
- `4` — ошибка GitHub CLI.

## Важные инварианты

- Скрипт выбирает не «лучшую по смыслу» задачу, а задачу по формальной policy.
- Неоднозначность разрешается детерминированно, а не эвристически.
- Любая structural inconsistency задачи делает её неисполняемой.
- Канонический workflow status живёт только в GitHub Project `Status` и выражается как `ready`, `in_progress`, `review`, `in_testing` или `done`.
